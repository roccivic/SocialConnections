Project Definition

Description

The project's name is College Social Connections. It will consist of a smart phone application and a PHP website. The main goal of this project is to substitute the sheets of paper that lecturers use all the time for taking attendance, grading students, etc. Thus, making life easier for them by doing everyday, work related, tasks on their smart phones or the website instead. Students should also be able to view grades and their attendance on their smart phones or the website that will be provided.

The purpose of the project

The purpose of this project is to make an easier way for lecturers to take attendance in all of their modules. In every college, attendance is very important to take note of. Student grants can be affected by their attendance to lectures and marks can also be assigned based on attendance. The current method is very out-dated where a piece of paper is sent around the room and is signed by the students. This led to the idea of creating a quicker and modern way of taking attendance. The information that will be gathered will be easily gathered and aggregated. It will also make it easier to access information regarding attendance and continual assessment results through an application on smartphones or through a web app. The current model of taking attendance is very slow and at times unreliable. It has to be considered that the application is not a necessity but the time it would save from each lecture makes it a productive application.

Goals of the project

The primary goal for the development of this application is to make it easier and quicker to take attendance of a class group. It is important the main goal for this project, to develop an application for attendance, is fully accomplished. We also want to make it possible for lecturers send results to individual students. To see if the application successfully meets the goal of the project we will test the application. A lecturer will attempt to take attendance through the face recognition option on a smartphone, any students not picked up by facial recognition, or that were absent, in the picture will have their names returned to the application. If there are any people that were not recognised, this can be remedied here. The lecturer, once logged in, can send results to the students. The lecturer can see the percentage of any students overall attendance of the module s/he teaches. The students can log in and will be alerted of new results available. Students can also see their previous results and their attendance percentage per module.

Client, Customer and other Stakeholders

Client

The client for this project, if it wwas to be used, would be Cork Institute of Technology. When the application is completed it will be possible to expand it to different mobile platforms, different languages and different colleges anywhere in the world.

The customer

There are four types of customers that will use this application. Lecturers, students and head of departments that are part of Cork Institute of Technology. Also it is predicted that this application will be useful to the grant office as some grants are dependent on attendance of the student. For the application to be considered complete, all the customers should have a functional and useful application.

Other stakeholders

Lecturers of students - Other lecturers of students may want to see what a student of theirs attendance is like in other modules the student is enrolled in.

The Users of the Product

User name: Student (Key user)
User role: A passive user. The application will be used mostly for viewing information about attendance, grades and notifications about CA results and updates about modules.
Subject Matter Experience: Master. All the data that is available to the student is directly connected to them e.g. attendance and results, and they will have a knowledge of all this data.
Technological Experience: Novice. The student will have no working knowledge of the back end of this application.
Other User Characteristics: The student will need to have the ability to either use a web browser or android application. The stock application and web app will be in English so the student will need to be able to comprehend English.

User name: Erasmus Student (Key user)
User role: A passive user. The application will be used mostly for viewing information about attendance, grades and notifications about CA results and updates about modules.
Subject Matter Experience: Master. All the data that is available to the student is directly connected to them e.g. attendance and results, and they will have a knowledge of all this data.
Technological Experience: Novice. The student will have no working knowledge of the back end of this application
Other User Characteristics: The student will need to have the ability to either use a web browser or android application. There will be an option to choose a language but the student will need to able to navigate the menus to show the application in their native language

User Name: Lecturer (Key user)
User Role: A lecturer will add data to the system e.g. attendance and results. The lecturer will also be able to view all students and their results and attendance to their module and other modules the students are attending.
Subject Matter Experience: Master. The lecturer will have knowledge of all the data as they will be contributing data to the system.
Technological Experience: Mid-range knowledge. The lecturer will not need to know how the system works but it is important for them to understand how to add, review, and modify data on the system to use the application fully.
Other User Characteristics: The lecturer will need to have the ability to either use a web browser or android application. The application will be quite complex for the lecturer if they want to see information about students’ attendance or results in other modules, so a good working knowledge of the application is crucial to gain full access to the system. The lecturer will need to able to understand how to upload information through the application or web app for all other users to view.

User Name: Head of department (Unimportant user)
User Role: A head of department will be able to view data about any student on the system.
Subject Matter Experience: Master. A head of department will have knowledge of all the data as it has to do with their department.
Technological Experience: Novice. A head of department will not need to have any working knowledge of the back end of the application or web app.
Other User Characteristics: The head of a department will need to have the ability to either use a web browser or android application. The application will be quite complex for the head of a department if they want to see information about students’ attendance or results in certain modules, so a good working knowledge of the application is crucial to gain full access to the system.

User Name: Grant Office (Secondary user)
User Role: A member of the grant office will view only certain students that are availing of the grant to determine how much of the grant a student will be receiving.
Subject Matter Experience: Master. A member of the grant office will have knowledge and access to all the data about every student and will have to understand how that data is accessed.
Technological Experience: Master. A member of the grant office will need to have a working knowledge of the system to extract the data they need on each student.
Other User Characteristics: The grant office will primarily be using the web app or have direct access to the data about attendance.

User Name: Administrators (Key user)
User Role: Administrators will have the ability to add, delete and modify records of students. Administrators could be head of departments, lectures and staff from the administration office in CIT.
Subject Matter Experience: Master. An administrator will have access to all data and have the ability to modify and view any records.
Technological Experience: Master. An administrator will need to have full understanding of how all the data is stored so because they have access over all data stored so any error on their part may massive consequences on the whole system.
Other User Characteristics: Administrators will need knowledge of PHPmyAdmin to edit any data on the system. Administrators will not need working knowledge of how the android application works, however they may need to know how to use the web app to view some data.

User Name: Developers (Key user)
User Role: Administrators will have the ability to add, delete and modify records of students. They can also add more modules, lecturers and administrators to the databases.
Subject Matter Experience: Master. A developer will have access to all data and have the ability to add, modify and view any records. They will also be able to alter the android application.
Technological Experience: Master. A developer has access to the whole system and can make changes as they wish, if they are confirmed by the developers.
Other User Characteristics: Developers will need knowledge of PHPmyAdmin to edit any data on the database system. Developers will need knowledge on how the application works and how it is coded if any changes need to be made.

Requirements

Functional Requirements

Using both the android app or the web interface, the student will be able to:

  • Login into his account, using either his student id or email in combination with a valid password
  • Check their overall attendance for each module that he is enrolled in
  • Check their attendance for labs in each module that he is enrolled in
  • Check their attendance for lectures in each module that he is enrolled in
  • View their results for CA assesments that have been published by the lecturer
  • Leave a note for the lecturer of a module for which he has missed a class
  • Get a link to notes posted on Dropbox by a lecturer

Using either the android app or the web interface, the student will be able to:

  • Post notes to Dropbox
  • View/post on twitter wall

Using the web app, the lecturer will be able to:

  • Create classes by adding student details one by one
  • Upload a class list of students whose attendance will be recorded
  • Add/Update/Delete members within a class

Using the android app, the lecturer will be able to:

  • Take pictures of students and upload them to take attendance
  • Manually take attendance of students from classes s/he teaches (in case of failed facial recognition
  • Post notes on Dropbox
  • View/post on twitter wall

Using both the android app and the web interface, the lecturer will be able to:

  • Login into his account, using either his student id or email in combination with a valid password
  • View attendence of students from classes that he teaches (needs to be expanded from specification document)
  • View attendance of students in modules besides their own
  • Post CA assesments results for modules that he teaches

Using the web app, the administrator will be able to:

  • Add a new department to the system
  • Compare attendance levels between different departments
  • Set threshold that defines poor attendance for a module
  • Set threshold that defines poor attendance for labs in a module
  • Add/Edit/Delete a lecturer
  • See how much students who are grant aided to are owed

The system will be able to:

  • E-mail attendance reminder to students weekly
  • Notify the lecturer about new posts on twitter
  • Notify the student about new posts on twitter
  • Notify the student about new CA results
  • Notify the student about new notes uploded to Dropbox
  • E-mail the head of department if a module is poorly attended

Non Functional Requirements

Security

  • An anonymous user shall not have any access to the system, with the exception of the greeting page and/or the login form
  • A student shall not have any access to features that require lecturer/administrator access
  • A lecturer shall not have any access to features that require administrator access
  • No user shall be able to execute arbitrary database queries (SQL injection)
  • The system shall limit requests the number of requests to the login functionality to a maximum of 3 requests per minute in order to prevent brute force attacks

Look and Feel

  • The interface will be minimalistic to ensure ease of use
  • The layout of the web app and the android application will be based on each other
  • The same experience will be available to all users of the application

Compatibility

Web app

The web app shall be compatibile with any browser that supports html 5 and javascript. These include following web browsers:

  • Firefox 3+
  • Chrome 4+
  • Safari 4+
  • Opera 10+
  • Internet Explorer 8+

The web app shall be able to run on a web server with Linux kernal 2.6+ that is running Apache 2.2+.

Mobile app

The mobile app shall be compatibile with devices running Android 2.2+ (API level 8+).

Reliability

  • The system shall issue a response to any request within 30 seconds
  • The system shall have a minimum uptime of 98% per year
  • In the event of a catastrophic failure, it should be possible to restore the system to a previous state that is no more than 7 days old

Maintainability

The source code of the system shall be clearly and effectively organised in order to allow new contributors to familiarise themselves with it after 7 days of training.

The source code shall contain no less than 12% of inline comments.

All developers will need a working knowledge so issues can be dealt with immediately

Extensibility

  • The system should be designed using Object-Oriented practices which will allow the maintainer to add new features easily
  • The documentation should be comprehensive in order to help the maintainer extend the functionality

Reusability

  • A different college should be able to personalize the system with very little changes
  • A college from a different country should easily be able to translate the system
  • The mobile application should be easily ported to other mobile operating systems

Resource Utilization

  • The android app should be no bigger than 10MB
  • The web server shall not utilise more than 1GB of memory while operating

Glossary

Admin

A type of User in the Social Connections system. An Admin may be a Lecturer or another person employed by the College, who is in charge of administering the Social Connections system.

Attendance

A measure of how often a Student is present in a Module. Calculated as follows:
Number of Classes where a Student was present divided by the total number of Classes.

CA / Continuous Assesment

An ongoing evaluation of a Student's progress in a Module. Accounts for the full grade or part of the grade that the Student receives upon completing a Module.

Class

A collection of Students that are attending the same Course and are in the same year of this Course. E.g: The Students that are taking the "Software Development" Course and are in the third year of this Course can be referred to as the "DCOM3" Class.
Alternatively, a Class can also be referred to as a Lecture or a Lab taking place on a specific day, at a specific time.

College

An institution providing higher education, specialized professional or vocational training.

Course

A programme composed of several Modules that takes place over several years that lead a successful student to obtain a degree.

Department

A logical part of the College. Departments are used to separate the college into smaller parts whose purpose is to teach related disciplines. E.g: Department of Business Studies, Department of Computing, Department of Tourism and Hospitality, etc...

Erasmus

A programme throughout Europe for Students living abroad for either one or two Semesters. It's goal is to improve the Students language and expose them to a foreign language.

Final Exam

An official examination carried out at the end of a Semester for one or more Modules. It determines the remaining part of a Student's grade, after taking CA into account.

Grant-aided Student

A Student that receives financial assistance from the government while attenting a Course at the College. The conditions for receiving such assistance generally entail that the Student is activel attending most Classes.

Group

A collection of Students that are taking the same Module. A group can be made up of Students from a Class (i.e: DCOM3), but it can also include Students that have previously failed a Module and are taking it for a second or subsequent time. A Group can also include Students from an exchange program, such as Erasmus.

Head of Department

A Lecturer that is responsible for the management of a Department's activities.

Lab

A Class taking place in a classroom that is especially adapted to allow Students to carry out pratical work.

Lecture

A Class taking place in a classroom, where the Lecturer delivers content to the Students. It is characterised by the absense of practical work.

Lecturer

A person employed by the College to deliver Classes, as well as a type of User in the Social Connections system.

Mobile App

An interface to the Social Connections system that can be accessed using a native app that can be installed on an Android smartphone or tablet.

Module

A series of Classes, optionally followed by a Final Exam that takes place over a Semester. A Module is attended by a Group of Students and is delivered by one or more Lecturers.

Semester

A measure of time that defines the length of a Module. It is generally 13 weeks.

Student

A person that attends a Course at the College, as well as a type of User in the Social Connections system.

User

A person that uses the Web App or the Mobile App to perform one of the tasks defined in the "functional requirements" section of this document.

Web App

An interface to the Social Connections system that can be accessed using a web browser.
