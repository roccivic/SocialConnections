Project Definition

Description

The project's name is College Social Connections. It will consist of a smart phone application and a PHP website with Joomla CMS. The main goal of this project is to substitute the sheets of paper that lecturers use all the time for taking attendance, grading students, etc. Thus, making life easier for them by doing everyday, work related, tasks on their smart phones or the website instead. Students should also be able to view grades and their attendance on their smart phones or the website that will be provided.

Project Schedule

We are given 2 months to achieve that goal. The project's start date is 28/01/13 and the deadline is on the 29/03/13.

We are going to take up an agile approach. Our highest priority will be to satisfy the “customer” and meet the requirements. Our plan is to have regular team meetings and work closely with our supervisor.

Requirements specification phase

Deadline: 8 February 2013

We will find out who the stakeholders are, and their interactions with the system. Gathering requirements and documenting them is the next step. We will develop Use Case diagrams or different for better understanding of the problem. They will also be used as test cases when the system is developed.

Design phase

Deadline: 1 March 2013

Our objective is to make a simple, efficient, flexible, maintainable, secure, reliable and object-oriented design. We will used diagrams, such as UML, to illustrate the design and interactions between classes.

Implementation phase

Deadline: 22 March 2013

We are going to be using git as our source code management system. Our git repository is hosted at https://github.com/roccivic/SocialConnections/.

Testing and Further Developments

Deadline: 27 March 2013

We are going to take up a distructive approach, meaning that our goal, while testing, will be to prove that the system fails. After we have found the failings we will attempt to fix them.

Documentation

Deadline: 29 March 2013

The documentation of this project will be an ongoing process during all the phases.

Requirements

Functional Requirements

Using both the android app and the web interface, the student will be able to:

  • Login into his account, using either his student id or email in combination with a valid password
  • Check his overall attendance for each module that he is enrolled in
  • Check his attendance for labs in each module that he is enrolled in
  • Check his attendance for lectures in each module that he is enrolled in
  • View his results for CA assesments that have been published by the lecturer
  • Leave a note for the lecturer of a module for which he has missed a class

Using the android app, the lecturer will be able to:

  • Post notes on Dropbox
  • Take pictures of students and upload them

Using both the android app and the web interface, the lecturer will be able to:

  • Login into his account, using either his student id or email in combination with a valid password
  • Take attendence of students for lectures and labs that he teaches
  • View attendence of students from classes that he teaches
  • View twitter wall
  • Post on twitter wall
  • Post CA assesments results for modules that he teaches

The system will be able to:

  • E-mail attendance reminder to students weekly
  • Alert the lecturer when there are new posts on twitter
  • Alert the student when there are new posts on twitter
  • Alert the student when there are new results

Non Functional Requirements

Security

  • An anonymous user shall not have any access to the system, with the exception of the greeting page and/or the login form
  • A student shall not have any access to features that require lecturer/administrator access
  • A lecturer shall not have any access to features that require administrator access
  • No user shall be able to execute arbitrary database queries (SQL injection)
  • The system shall limit requests the number of requests to the login functionality to a maximum of 3 requests per minute in order to prevent brute force attacks

Compatibility

Web app

The web app shall be compatibile with the following web browsers:

  • Firefox 3+
  • Chrome 4+
  • Safari 4+
  • Opera 10+
  • Internet Explorer 8+

Mobile app

The mobile app shall be compatibile with devices running Android 2.2+ (API level 8+).

Reliability

  • The system shall issue a response to any request within 30 seconds
  • The system shall have a minimum uptime of 98% per year
  • In the event of a catastrophic failure, it should be possible to restore the system to a previous state that is no more than 7 days old

Maintainability

The source code of the system shall be clearly and effectively organised in order to allow new contributors to familiarise themselves with it after 7 days of training.

The source code shall contain no less than 12% of inline comments.
